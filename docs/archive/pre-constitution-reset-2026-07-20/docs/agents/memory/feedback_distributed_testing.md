**Когда нужно прогнать ВСЮ суйту — делить на три машины, не гонять соло.**

1. Задача делится на троих.
2. **GreenBear — 6 ядер → ему ~40% нагрузки.**
3. **Linode — 1 ядро → туда самое ПРОСТОЕ/лёгкое** (быстрые юнит-тесты без тяжёлых
   фикстур), чтобы сервер успевал обрабатывать живые запросы наравне с тестами.
4. **bolt — 8 ядер → самая большая доля** (в т.ч. тяжёлый chef_battle, 441 тест).
5. Если GreenBear занят другим — **насрать, доделает потом**: дать ему чёткий
   приказ ЗАПОМНИТЬ, где остановился → выполнить свою долю тестов → потом
   продолжить. И так ВСЕГДА.

**Насыщать ресурсы.** Владелец мониторит: если у него CPU 49% и память 60% при
идущей суйте — значит либо тесты кончаются, либо машина недогружена. Цель ~100%.
NB: `--parallel 8` на Windows+Postgres часто **I/O-bound по базе** (воркеры ждут
Postgres, ядра простаивают → ~49% CPU) — соло-прогон машину не насыщает,
распределение и/или больше воркеров на I/O-ожидании поднимают утилизацию.

**Why:** 2026-07-18 я гонял всю суйту соло на своих 8 ядрах, забыв протокол
(потерян между сессиями). Владелец заметил недогруз (49% CPU / 60% mem) и напомнил
дословно. Это медленнее и не использует GB+Linode.

**How to apply:** шардить по приложениям/модулям (`manage.py test <labels>`), каждая
машина — против своей тестовой БД, результаты независимы, зелёными должны быть ВСЕ
три. Linode-долю держать лёгкой (проверять трафик перед стартом — [[reference-run-tests-server]]).
GB-долю отдавать приказом через CoWork с чекпоинтом его текущей задачи.
Помнить ловушку: `--parallel` на Windows маскирует traceback упавшего теста
(pickle) — при падении перегнать долю через `--parallel 1`.
